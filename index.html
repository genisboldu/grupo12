LAS 5 'V' DEL BIG DATA

Dimensión Descripción
VOLUMEN Indica la gran cantidad de datos que generan y almacenan grandes plataformas, cómo las redes sociales.

Una empresa que gestiona una red social, almacena petabytes de datos a través de sus publicaciones, videos, interacciones, y MySQL, es una base de datos que no está preparada para trabajar con estos volúmenes, ya que tiene problemas con la escalabilidad horizontal y gestión de almacenamiento distribuido.
VELOCIDAD Indica la velocidad en la que se crean y procesan los datos, un campo muy importante ya que estamos acostumbrados a tiempos de carga muy muy lentos.

Un ejemplo, sería un sistema de mensajería, como podría ser WhatsApp o telegram, en cuando lo enviamos, tiene que marcar al instante, enviado, recibido, y si es el caso, leído. MySQL, tiene retrasos, ya que necesita escribir por cada mensaje una fila y actualizar estados genera cuellos de botella si hay muchos usuarios, millones, enviando mensajes a la vez. La escalabilidad limitada seria otro problema.
VARIEDAD Indica la variedad de tipo de datos que se pueden procesar, cómo tablas, JSON, XML, imágenes, videos, texto.

Un ejemplo sería los datos que pude generar un móvil, como las fotos, que son en JPEG, un mensaje, en formato texto y la ubicación, que sería en formato JSON. MySQL, no procesa muy bien formatos JSON ni tampoco las imágenes, penaliza mucho en velocidad y capacidad, si guardamos datos muy grandes y requiere de herramientas externas para procesar datos no estructurados, como las imágenes y videos.
VERACIDAD Representa la consistencia y la calidad de los datos procesados, tratando problemas como datos incompletos o inconsistencia.

Un ejemplo sería la gestión de formularios, dónde un usuario puede enviar información con un correo no válido, o ya registrado, o palabras con errores tipográficos. MySQL, no está preparado para validar correos electrónicos con formatos correctos. MySQL, si tiene que corregir errores tipográficos o completar información, tiene que hacer múltiples consultas, o utilizar scripts externos, y eso le penaliza mucho en tiempos de procesamiento.
VALOR Indica todo el uso que le podemos extraer de los datos, su función es convertir los datos en información de utilidad.

Un ejemplo sería el historial de compra en Amazon, para recomendar productos de interés o parecidos. MySQL no puede procesar patrones o comportamientos, requiere de herramientas externas para identificar estos patrones, que pude ser muy lento y penalizar en tiempos.

Hadoop

2. Dos distribuciones de Hadoop

Hadoop es un framework de software de código abierto diseñado específicamente para el procesamiento distribuido y almacenamiento de grandes volúmenes de datos de manera eficiente. Este sistema se basa en el uso de clusters, donde múltiples nodos trabajan de forma coordinada para gestionar datos a gran escala. Se describen dos de las distribuciones más populares: Cloudera Distribution of Hadoop (CDH) y Hortonworks Data Platform (HDP), junto con los módulos que incluyen y sus funciones principales.

Cloudera Distribution of Hadoop (CDH)

Cloudera es una de las empresas pioneras en el ecosistema de Hadoop y ofrece CDH como una distribución diseñada específicamente para entornos empresariales. Esta plataforma no solo implementa las capacidades básicas de Hadoop, sino que también incluye herramientas adicionales para análisis, gestión de datos y monitorización. El enfoque de CDH está orientado a facilitar el manejo de grandes volúmenes de información, proporcionando soporte robusto y características avanzadas para el análisis de datos en tiempo real.

Módulos incluidos en CDH y sus funciones:

  • HDFS (Hadoop Distributed File System): Este módulo es el núcleo del almacenamiento distribuido en CDH. HDFS divide grandes archivos en bloques y los distribuye entre los nodos del cluster, garantizando redundancia y fiabilidad. Esto asegura que los datos se mantengan accesibles incluso en caso de fallos en los nodos.
  • YARN (Yet Another Resource Negotiator): Este gestor de recursos distribuidos organiza y asigna de manera eficiente la memoria y el procesamiento a las diferentes aplicaciones que se ejecutan en el cluster. YARN permite que múltiples aplicaciones compartan los recursos del sistema de manera equilibrada.
  • Apache Hive: Hive es una herramienta de consulta SQL que permite interactuar con los datos almacenados en HDFS. Es ideal para realizar análisis y consultas estructuradas utilizando un lenguaje similar al SQL, facilitando el acceso a los datos para analistas y desarrolladores.
  • Apache Impala: Impala es un motor SQL que complementa a Hive, pero se destaca por su capacidad de realizar análisis de datos en tiempo real. Es ampliamente utilizado en escenarios donde la velocidad es crucial, como en paneles de control interactivos y análisis ad-hoc.
  • Apache Spark: Spark es un framework para el procesamiento de datos en memoria. Este módulo es esencial en CDH para realizar análisis rápidos y ejecutar tareas de aprendizaje automático, procesamiento de flujos y consultas interactivas.
  • Cloudera Manager: Esta plataforma proporciona una interfaz gráfica para la configuración, monitorización y gestión de clusters Hadoop. Con Cloudera Manager, los administradores pueden supervisar el rendimiento, identificar cuellos de botella y solucionar problemas de manera centralizada.

Hortonworks Data Platform (HDP)

Hortonworks desarrolló HDP como una distribución completamente de código abierto, orientada a empresas que buscan implementar soluciones Hadoop sin depender de software propietario. HDP se centra en la integración y la simplicidad de uso, ofreciendo herramientas para el procesamiento y la gestión de datos que cumplen con los estándares empresariales. Además, su enfoque en la comunidad y en la interoperabilidad lo convierte en una opción atractiva para muchas organizaciones.

Módulos incluidos en HDP y sus funciones:

  • HDFS y YARN: Al igual que en CDH, HDFS y YARN forman el núcleo del almacenamiento y la gestión de recursos. HDFS se encarga de almacenar los datos de manera distribuida en todo el cluster, mientras que YARN asigna y gestiona los recursos necesarios para ejecutar aplicaciones de manera eficiente.
  • Apache Tez: Tez es una alternativa más avanzada a MapReduce, diseñada para ejecutar flujos de trabajo complejos de manera más rápida y eficiente. Es especialmente útil para tareas como el procesamiento en serie de grandes conjuntos de datos.
  • Apache Ambari: Esta herramienta facilita la configuración, monitorización y gestión de los clusters de Hadoop. Ambari ofrece una interfaz gráfica intuitiva que permite a los administradores instalar nuevos nodos, monitorear el rendimiento del cluster y realizar ajustes según sea necesario.
  • Apache Ranger: Ranger es un módulo de seguridad que proporciona control de acceso centralizado para los datos almacenados en el cluster. Con Ranger, las empresas pueden definir políticas detalladas para garantizar que solo los usuarios autorizados puedan acceder o modificar ciertos datos.
  • Apache Kafka: Este sistema se especializa en la gestión de flujos de datos en tiempo real. Kafka es ideal para casos de uso donde se necesita transmitir y procesar grandes volúmenes de datos en tiempo real, como en aplicaciones de Internet de las Cosas (IoT) o en sistemas de monitoreo continuo.
  • Apache Atlas: Este módulo es una herramienta de gestión de metadatos que permite rastrear el linaje de los datos en todo el ecosistema. Con Atlas, los usuarios pueden entender cómo se generan, transforman y consumen los datos, facilitando la auditoría y el cumplimiento normativo.

3. Modos de ejecución de Hadoop

Hadoop puede ejecutarse en tres modos diferentes según el entorno y las necesidades específicas del usuario. Estos modos son: modo local (Standalone), modo pseudo-distribuido y modo distribuido.

Modo local (Standalone)

En este modo, Hadoop opera en un único nodo, y es el modo predeterminado tras su instalación. En este caso, no se utiliza HDFS ni YARN, ya que todos los procesos se ejecutan localmente en el sistema de archivos del nodo. Esto hace que sea una configuración simple y rápida para iniciar, ideal para realizar pruebas, desarrollar aplicaciones o depurar código sin la complejidad de un entorno distribuido. No requiere configuraciones avanzadas ni múltiples nodos, por lo que es ampliamente utilizado por los desarrolladores durante las primeras etapas del desarrollo.

Modo pseudo-distribuido

El modo pseudo-distribuido simula un entorno distribuido dentro de un único nodo. En este caso, los principales servicios de Hadoop, como HDFS, YARN y MapReduce, se configuran para ejecutarse como si estuvieran distribuidos en un cluster, aunque físicamente están en un solo sistema. Este modo es útil para quienes desean probar aplicaciones en un entorno que emule las condiciones de un cluster real sin necesidad de contar con múltiples servidores físicos. Permite identificar cómo se comportarían las aplicaciones y los servicios en un entorno distribuido antes de desplegarlos en producción.

Modo distribuido

Por último, el modo distribuido es la configuración utilizada en entornos de producción. En este modo, Hadoop se implementa en un cluster completo que puede incluir múltiples nodos físicos o virtuales. Aquí, HDFS se encarga de distribuir los datos entre los diferentes nodos del cluster, lo que garantiza un almacenamiento eficiente y redundante. Por su parte, YARN gestiona los recursos del cluster, asegurando que las aplicaciones reciban la cantidad adecuada de memoria y capacidad de procesamiento. Este modo es esencial para procesar grandes volúmenes de datos y aprovechar la capacidad de escalabilidad horizontal que caracteriza a Hadoop. Es el modo más complejo de configurar y mantener, pero también el más poderoso y eficiente para el procesamiento de datos a gran escala.

4. Roles de los nodos en YARN

Nodo Master (ResourceManager)

El ResourceManager es el nodo principal y central del clúster en YARN, encargado de gestionar todos los recursos disponibles en el sistema. Su función principal es decidir cómo se asignan estos recursos entre las distintas aplicaciones que se ejecutan en el clúster. Para ello, analiza las solicitudes enviadas por las aplicaciones a través de sus respectivos ApplicationMasters y determina la cantidad de recursos que se pueden asignar en función de la disponibilidad y las políticas configuradas (como prioridades, colas y límites de uso).

Además de gestionar los recursos, el ResourceManager tiene la tarea de realizar un seguimiento constante de los trabajos que están en ejecución. Esto implica supervisar su estado y garantizar que los recursos asignados se utilicen de manera eficiente. También es responsable de la planificación de las tareas, lo que incluye decidir cuándo y dónde se deben ejecutar los contenedores para maximizar el rendimiento del clúster y evitar cuellos de botella. Al centralizar la gestión de recursos, el ResourceManager permite optimizar el uso del hardware y facilita la escalabilidad del sistema.

Nodo Worker (NodeManager)

Cada nodo físico o virtual en un clúster YARN incluye un NodeManager, que actúa como el componente local encargado de gestionar los recursos específicos de ese nodo. Entre los recursos que administra se encuentran la capacidad de procesamiento (CPU), la memoria y el almacenamiento local. El NodeManager se asegura de que estos recursos se asignen adecuadamente a las tareas que se ejecutan en el nodo.

Una de sus principales funciones es ejecutar los contenedores, que son las unidades de trabajo donde se llevan a cabo las tareas asignadas por las aplicaciones. Cada contenedor opera dentro de los límites de recursos establecidos por el NodeManager, garantizando que no haya interferencias entre las tareas.

El NodeManager también comunica de manera constante con el ResourceManager, proporcionando información actualizada sobre el estado de los recursos en su nodo. Esto permite al ResourceManager tomar decisiones informadas sobre cómo distribuir las próximas tareas en el clúster. Si un nodo experimenta problemas (como fallos en hardware o falta de recursos), el NodeManager informa de inmediato al ResourceManager para que este pueda reasignar las tareas y mantener la estabilidad del sistema.

ApplicationMaster

El ApplicationMaster es un componente específico para cada aplicación que se ejecuta en el clúster de YARN. Su papel es coordinar todas las tareas relacionadas con esa aplicación particular. Cuando una aplicación se inicia, el ApplicationMaster solicita recursos al ResourceManager en nombre de la aplicación, especificando los requisitos necesarios para completar su trabajo.

Una vez que el ResourceManager asigna los recursos, el ApplicationMaster supervisa el progreso de las tareas que se ejecutan en los contenedores distribuidos en diferentes nodos del clúster. Esto incluye gestionar el lanzamiento de contenedores, asegurarse de que las tareas se completen correctamente y manejar situaciones de error, como reiniciar tareas fallidas o reasignarlas a otros nodos.

Además, el ApplicationMaster actúa como intermediario entre la aplicación y los nodos del clúster, recopilando y reportando información sobre el estado y los resultados de las tareas al usuario o al sistema que solicitó la ejecución. Su existencia permite que cada aplicación tenga un control más personalizado y adaptado a sus necesidades específicas dentro del entorno distribuido.

5. Diferencias entre MySQL y HiveSQL

Una diferencia entre MySQL y HiveSQL está en la creación de tablas. En MySQL, es necesario definir detalladamente el esquema de la tabla, incluyendo el tipado de los datos y los índices entre tablas.

En HiveSQL, el tipado de datos es más flexible y permite la definición de tablas externas.

MySQL permite operaciones como INSERT, UPDATE o DELETE. Por el contrario, HiveSQL solo permite INSERT. Para modificar los datos en HiveSQL, normalmente se crean nuevas tablas o se sobrescriben los datos.

MySQL cuenta con una gran variedad de funciones integradas. HiveSQL tiene menos funciones integradas, aunque se puede ampliar mediante funciones creadas por el usuario.

MySQL utiliza índices entre tablas para la optimización de consultas. HiveSQL no usa índices tradicionales; en su lugar, depende de otras funciones, como estructuras de archivo, para optimizar las consultas.

El tipado de datos es más amplio en MySQL. HiveSQL no tiene un rango tan extenso de tipos de datos, ya que está diseñado para manejar grandes volúmenes de datos. Los tipos más comunes en HiveSQL son STRING, DOUBLE, ARRAY, MAP, entre otros.

1.Diferencia entre un gráfico, un dashboard y una story

  • Los gráficos son como se representan visualmente los datos en un formato en concreto, como mapas, diagramas...
  • Un dashboard sería un conjunto de gráficos, que se puedes aplicar filtros y elementos agrupados en una sola vista. Permiten analizar varias perspectivas de datos y explorar relaciones entre diferentes gráficos.
  • Una story es una secuencia de dashboards y gráficos, ordenados para explicar un análisis y conclusiones entre ellos. Una story facilita al usuario navegar de forma lineal en dashboards y gráficos.

2.Tres conexiones dónde Tableau puede extraer datos

  • Permite conectarnos a Microsoft Excel, con archivos de hojas de cálculo.
  • Permite conexiones a bases de datos, como por ejemplo en SQL Server, para grandes volúmenes de datos.
  • Permite conexiones a servicios en la nube, como Amazon Redshift.

3.Conexiones en Tableau

En Tableau, cuando trabajamos con datos desde un servidor, tenemos dos modos principales de conexión: conexión en vivo y extracción. Estos métodos determinan cómo Tableau accede y utiliza los datos de la fuente, y cada uno tiene sus propias ventajas y limitaciones que se adaptan a diferentes necesidades de análisis.

Conexión en vivo

La conexión en vivo permite que Tableau se conecte directamente a la fuente de datos alojada en el servidor. Esto significa que cada vez que interactuamos con un dashboard, realizamos filtros o actualizamos una vista, Tableau envía una consulta directamente a la base de datos para obtener la información más actualizada.

Ventajas

  • Siempre se trabaja con datos en tiempo real, lo cual es esencial en entornos donde los datos cambian constantemente, como en sistemas de ventas en línea, seguimiento de inventarios o análisis financiero en tiempo real.
  • No es necesario realizar actualizaciones manuales porque los datos se sincronizan directamente desde la fuente.

Desventajas

  • El rendimiento puede depender de la capacidad del servidor donde están alojados los datos y de la velocidad de la red. Si el servidor es lento o está sobrecargado, el tiempo de respuesta puede ser más largo.
  • El acceso constante al servidor puede aumentar la carga de trabajo en la fuente de datos, especialmente si hay múltiples usuarios realizando consultas simultáneamente.
  • Requiere una conexión estable a la base de datos; si la conexión se interrumpe, no será posible interactuar con los datos.

Ejemplo práctico

En una empresa minorista que realiza un seguimiento en tiempo real de las ventas en sus sucursales, una conexión en vivo garantiza que el equipo de operaciones pueda monitorear instantáneamente las transacciones y ajustar la logística según la demanda.

Extracción

En este modo, Tableau realiza una copia de los datos de la fuente y los almacena localmente en un archivo de extracción, generalmente en formato .hyper. Esta copia permite trabajar con los datos sin necesidad de mantener una conexión constante con el servidor.

Ventajas

  • Ofrece un mejor rendimiento al realizar consultas y actualizaciones en Tableau, ya que los datos se almacenan de forma local y no se depende de la capacidad de respuesta del servidor.
  • Reduce la carga en la fuente de datos, ya que las consultas no se envían directamente al servidor una vez que se ha creado la extracción.
  • Permite trabajar con los datos incluso sin conexión a la base de datos, lo cual es útil en entornos donde la conectividad puede no estar garantizada o donde se necesita movilidad.
  • Puede filtrar y resumir datos antes de extraerlos, lo que ayuda a reducir el tamaño de los conjuntos de datos grandes y agiliza el análisis.

Desventajas

  • Los datos no están en tiempo real. Para reflejar los cambios realizados en la fuente, es necesario actualizar manualmente la extracción.
  • Si los datos originales son muy grandes, la extracción inicial puede tardar más tiempo y requerir más almacenamiento local.

Ejemplo práctico

En una empresa que realiza análisis de datos históricos sobre ventas anuales, utilizar una extracción facilita realizar análisis detallados sin depender de la conectividad con el servidor, ya que los datos no cambian con frecuencia.

4.Tipos de Gráficos

Gráfico de Barras

Representa variables usando barras horizontales o verticales.
Ejemplo: comparar las concentraciones medias de contaminantes (variable numérica) en distintos puntos geográficos de medición (categoría).

Gráfico de Líneas

Representa la relación de dos variables cuantitativas continuas mediante una línea.
Ejemplo: ver la variación de la concentración media de un contaminante a lo largo del tiempo (series temporales y tendencias).

Gráfico de Dispersión (Scatter Plot)

Representa la relación de dos variables cuantitativas mediante puntos.
Ejemplo: ver si existe correlación entre el peso y la altura de las personas (análisis de regresión).

Gráfico Circular

Muestra porciones de distintas categorías dentro de un conjunto de datos.
Ejemplo: estudiar las categorías de películas más vistas en una plataforma de streaming.
